# 知识质量检查 Reference

Use this reference when the user asks for a full weekly lint, 知识质量检查, 知识质量报告, solution absorption review, or project backflow review.

## 默认报告路径

Write the report to `.llm-wiki/reports/知识质量检查-YYYYMMDD.md` using the local date.

Create `.llm-wiki/reports/` if it does not exist.

## 默认执行方式

先自动执行，再写报告。

执行前先按 `05_Templates/笔记整理归档SOP.md` 确认目录身份：

- `02_Sources`：原始输入和未整理 QA。
- `03_Notes`：长期稳定判断。
- `04_Projects`：项目现场、业务规则、运维规则、内容生产。
- `docs/solutions`：CE 方法资产，不是原始 QA。
- `.llm-wiki`：治理、状态、报告、质量检查。

可以自动做：

- source 里的稳定判断写入已有 `03_Notes`
- 给 note 补 `source_refs`
- 把已吸收 source 标记为 `llm_status: absorbed`
- 从 solution 抽一句长期原则补进已有 note
- 从 project 或已发布文章补引用到已有 note，但不追改受保护发布稿正文
- 小范围修正 SOP / checklist 的当前规则口径

只报告、不自动做：

- 合并或拆分 note
- 新建、升级、拆分 domain
- 删除文档或模板
- 大范围重写 project 事实
- 编辑 `published_url` 非空的受保护发布稿正文
- 移动大目录

## 六个检查面

### 1. Source intake

Review `02_Sources` and `02_Sources/LLM Wiki 处理台.base` for candidates that need routing, absorption, or review.

Report:

- high-value sources to absorb next
- stale or low-value backlog
- sources missing `llm_status`, `llm_domain`, `ddc`, or `llm_note`
- sources that need human judgment before absorption

### 2. Wiki body

Review `03_Notes`, the target domain MOC, and the domain's `Wiki Index` / `Wiki Log` when present.

Report:

- duplicated definitions
- pages that should merge or split
- stale judgments
- summary pages that should update an existing page instead of creating a new one
- pages missing `source_refs` when the source chain is known

### 3. Query write-back

Review repeated questions, QA notes, chat-derived insights, and write-back logs when present.

Report:

- repeated questions that deserve stable pages
- durable judgments discussed but not written back
- query-derived notes that need source references
- write-back candidates that should remain transient

### 4. Solution absorption

Treat `docs/solutions` as reusable method assets maintained by CE workflows.

Do not copy whole solution retrospectives into `02_Sources` or `03_Notes`.

Classify sampled solution entries as:

- `write to 03_Notes`: long-term principle, boundary, repeated judgment, or reusable conceptual distinction
- `update 04_Projects`: project operating fact, current service state, business rule, or workflow entry point changed
- `no absorption needed`: one-time operation log, already covered elsewhere, or no durable judgment
- `missing source_refs`: an existing note covers the idea but cannot trace back to the solution

### 5. Project backflow

Treat `04_Projects` as project-local operating truth, not as a dumping ground for evergreen knowledge.

Classify sampled project material as:

- `keep in project`: project fact, current state, runbook, published artifact, or active project document
- `write to 03_Notes`: stable principle, judgment, boundary, or reusable model
- `generate docs/solutions`: repeatable workflow or reusable problem pattern
- `refresh project`: project fact appears stale and should be verified or updated

Published article protection applies only to `04_Projects/AI_Media` and the Zhihu output directory when frontmatter has non-empty `published_url`. Protected published bodies should not be edited directly; absorb new judgments through later articles, errata, supplements, or knowledge cards.

Do not expand this protection to all `04_Projects`: business rules, operations rules, project descriptions, deployment docs, and current-state docs should be verified and updated when facts changed.

### 6. Topology

Review `.llm-wiki/domains.json`, MOCs, Indexes, Logs, and recurring note clusters.

Report:

- parent domains that are too mixed
- child MOCs that behave like independent domains
- registry mismatch or missing domain entries
- promotion proposals
- operating files that no longer match the actual note graph

## 报告栏目

The user-facing report should be compact. Include these sections:

1. `已自动完成`
2. `需要确认`
3. `暂不处理 / 保留原位`
4. `下次候选`

Use the six check surfaces as internal classification only. If helpful, add a `来源类型` column with values like `source -> note`, `solution -> note`, `project -> note`, `topology`, or `template refresh`.

Decision labels:

- `已执行`: already changed files
- `需确认`: requires user or domain decision
- `保留原位`: intentionally not absorbed or changed
- `下次处理`: good next item, not handled in this pass

## 边界

- Prefer diagnosis over broad rewriting.
- Fix only tiny mechanical issues during lint unless the user explicitly asks for an edit pass.
- Do not create `_archived/` for stale docs; use git history when deletion is appropriate.
- Do not move large `04_Projects` trees during lint.
- Do not rename or restructure `04_Projects/AI_Media` during lint.
- Do not expand published-article protection to all project documents.
